# PRD：AEP（Agent Escrow Protocol）去中心化AI托管结算协议｜最终执行版
## 0. 文档说明
本文档为AEP针对 Cobo 黑客松赛道的最终执行PRD。项目精准对齐赛道评审核心：**CAW 是 Agent 经济从“玩具”走向“真实商业”的资金流底座**。AEP 解决了 Agent 自主交易中“付了钱不给货/给垃圾货”的致命痛点，通过 CAW Pact 实现资金托管，通过 CAW Recipe 实现条件结算，让 Agent 真正具备“管钱、花钱、赚钱”的经济自理能力与安全边界。
---
## 1. 场景贴合度：Agent 是经济主体，钱包是生命线
### 1.1 Agentic Commerce 的真实痛点
当前的 Agent 交易演示多为“一键支付（x402）”，这是机器行为，不是商业行为。真实的 Agentic Commerce 是异步的、有博弈的：Agent A 雇佣 Agent B 处理高价值数据，B 可能交付垃圾数据骗钱。
如果钱包只是个“付款插件”，Agent 必定被骗到破产。**Agent 要作为经济主体参与市场，前提是它的资金具备“履约条件锁定”与“质量不达标退款”的自卫能力。**
### 1.2 AEP 的场景定义
AEP 聚焦于**高价值、异步交付、需质量验收**的 A2A 雇佣场景（如：量化数据采购、代码审计、算力租赁）。
在这些场景中，Agent 不是单纯花钱，而是**“管钱（锁定预算）— 花钱（按条件解锁）— 赚钱（交付获结算）”**。CAW 钱包在此不是附属支付功能，而是 Agent 的“信托账户”，是交易能否达成的信任前提。
---
## 2. CAW 关键性：不可替代的托管执行引擎
### 2.1 为什么不用智能合约？CAW 是体验与安全的唯一解
传统 DApp 依赖智能合约托管资金，这在 Agent 高频交易场景下是灾难：
- **体验差**：12秒出块延迟，Agent 任务流转卡顿；
- **不安全**：资金转入合约，用户丧失控制权，合约漏洞导致资金归零；
- **不灵活**：风控规则写死在合约里，无法动态干预。
### 2.2 CAW Pact/Recipe：AEP 托管结算的底层心脏
AEP 的核心逻辑是“验收后才放款”，这个动作的执行者**必须是且只能是 CAW**。
- **CAW Pact 即托管池**：BuyerAgent 发起任务时，调用 CAW API 锁定资金（Pact 状态: Locked）。此时资金仍在 Buyer 控制域内，但被密码学契约约束，任何人（包括 Buyer）无法单方面挪用。
- **CAW Recipe 即放款指令**：当 Evaluator 规则校验通过，AEP Relayer 调用 CAW Recipe API，触发 P2P 极速结算（Pact 状态: Released，<500ms）；若校验失败，触发退款（Pact 状态: Refunded）。
- **不可替代性**：如果没有 CAW 的 Pact 机制，AEP 要么退回到不安全的合约托管，要么退回到没有资金保障的 x402 信用支付。CAW 是 AEP 实现安全与极速的物理基石。

## 3. 资金流程完整度：从触发到终结的三态生命周期
AEP 的资金流不是一笔转账，而是一个完整的**契约生命周期**。Demo 将完整展示以下流程：
| 阶段 | 触发条件 | CAW Pact 状态 | 资金归属 | 评委可见视觉脉冲 |
|------|----------|--------------|----------|-----------------|
| **1. 锁定预算** | BuyerAgent 悬赏发布 | **Locked** | 买方账户内锁定，不可动 | 🔵 蓝色锁定脉冲 |
| **2. 交付验证** | SellerAgent 提交数据 | Locked | 买方账户内锁定，不可动 | 🟡 黄色校验脉冲 |
| **3a. 极速结算** | 规则校验通过 | **Released** | 毫秒级划转至卖方 | 🟢 绿色结算脉冲 |
| **3b. 风险退款** | 规则校验失败/超时 | **Refunded** | 原路退回买方账户 | 🔴 红色退款警报 |

## 4. 核心架构：确定性校验保障资金安全
### 4.1 三层解耦架构
- **资金托管层（CAW MPC）**：执行 Pact 的锁定与 Recipe 的释放/退款，保障资金物理安全。
- **状态存证层（Foundry 合约）**：链上记录 Job 生命周期与 `JobSlashed` 作恶事件，不碰资金。
- **履约校验层（Go 规则主轨 + LLM 旁路）**：**规则拥有一票否决权**，是资金是否 Release 的唯一法理依据。LLM 仅生成报告，不操作资金。
### 4.2 架构克制（不画饼）
- **放弃纯 LLM 决策**：防范提示词注入盗取资金。
- **放弃合约资金池**：防范合约漏洞与资金挪用。
- **克制递归互雇**：MVP 强制 `depth_limit=1`（单级结算），杜绝多级嵌套带来的资金死锁，确保 Demo 稳定。

## 5. 核心功能说明（开发与演示依据）
### F1: 悬赏发布与资金锁定
- **简介**：BuyerAgent 发起异步雇佣任务，通过 CAW Pact 锁定悬赏资金，建立密码学托管契约，并在链上广播任务。
- **输入**：任务描述元数据、悬赏 Token 地址与金额、收款 Agent 白名单（可选）、Pact 超时区块高度。
- **输出**：CAW Pact ID（状态: Locked）、链上 `BountyPosted` 事件、AEP 系统 Job ID。
- **操作手顺**：
  1. BuyerAgent 构造任务参数与资金锁定条件；
  2. 调用 CAW API 创建 Pact，授权锁定指定额度资金；
  3. 获取返回的 `pact_id`；
  4. 调用 Foundry 合约 `postBounty(pact_id, metadata)`，将任务上链；
  5. SSE 大屏捕捉 `BountyPosted` 事件，展示 🔵 蓝色脉冲。
### F2: 身份准入与抢单
- **简介**：SellerAgent 监听链上事件，校验自身 FluxA 声誉是否达标，达标后抢夺任务独占执行权。
- **输入**：链上 `BountyPosted` 事件、SellerAgent 的 DID 标识。
- **输出**：FluxA 声誉校验结果、链上 `BountyClaimed` 事件、合约任务状态变更为 Claimed。
- **操作手顺**：
  1. SellerAgent 后端 WS/轮询监听到新悬赏；
  2. 携带 DID 调用 FluxA API 查询声誉分；
  3. 判断声誉分 ≥ 200（若 API 超时 3s 则降级放行）；
  4. 调用 Foundry 合约 `claimBounty(jobId)`；
  5. 合约校验身份并锁定状态（防并发抢单），触发 `BountyClaimed` 事件。
### F3: 双轨校验与裁决
- **简介**：对 SellerAgent 的交付物进行确定性规则硬校验（主轨决定资金去向）与 LLM 语义评估（旁路仅生成报告），防范提示词注入导致资金流失。
- **输入**：交付物 URI（如 IPFS Hash）、预期规则配置（如数据行数 ≥ 100, 必含字段）、任务原始描述。
- **输出**：Go 规则校验结果 / LLM 评估报告。最终裁决结果（Pass / Fail）。
- **操作手顺**：
  1. Seller 提交交付物 URI，调用合约 `submitResult(jobId, uri)`；
  2. EvaluatorAgent 监听事件，下载交付物；
  3. **主轨执行**：Go 程序执行确定性规则检查（JSON 解析、字段统计）；
  4. **旁路执行**：LLM 分析数据与任务描述的语义匹配度，生成可读评估报告；
  5. **终局裁决**：若 Go 规则全过则终裁 Pass；任一规则不过则终裁 Fail（规则一票否决，无视 LLM 结果）。
### F4: 条件结算 / 风险退款
- **简介**：根据 F3 的裁决结果，由 AEP Relayer 驱动 CAW 完成资金的生命周期终结（放款或退回）。
- **输入**：F3 裁决结果、Job ID、关联的 `pact_id`。
- **输出**：CAW Pact 状态变更（Released / Refunded）、链上 `JobCompleted` / `JobFailed` 事件。
- **操作手顺**：
  - **结算路径 (Pass)**：Relayer 监听到裁决 Pass -> 调用 Foundry `verifyJobResult(jobId, true)` 状态机变为 Completed -> Relayer 调用 CAW Recipe API 释放 Pact -> 资金 <500ms 划转至 Seller -> 🟢 绿色脉冲。
  - **退款路径 (Fail)**：Relayer 监听到裁决 Fail -> 调用 Foundry `verifyJobResult(jobId, false)` 状态机变为 Failed -> Relayer 调用 CAW API 触发 Pact Refund -> 资金退回 Buyer -> 🔴 红色警报。
### F5: 声誉惩罚与存证
- **简介**：对恶意交付触发退款的 Agent 进行链上存证记录，驱动生态自治惩罚。
- **输入**：退款触发事件、恶意交付的校验日志。
- **输出**：链上 `JobSlashed(sellerAddr, jobId)` 事件。
- **操作手顺**：
  1. Foundry 合约在状态流转至 Failed 时，自动 emit `JobSlashed` 事件；
  2. 外部声誉系统（FluxA / 未来的 ERC-8004）可通过订阅该事件，自主扣减作恶 Agent 的声誉分；
  3. 实现 AEP 协议与身份体系的解耦治理。

## 7. 风险边界说明（赛道硬性要求对齐）
本项目涉及资金托管与链上交互，严格声明以下边界与控制方式：
1. **资产边界**：Demo 全程运行于 **Base Sepolia 测试网**，使用无价值测试代币，绝无真实资产风险，无金融炒作属性。
2. **密钥控制权**：基于 CAW MPC 架构，**私钥从未以完整形式存在于内存或服务器**。Agent 仅持有受限的 API Token，无法提取资金，只能执行预定义的 Pact/Recipe 操作。
3. **资金操作边界**：资金只有三种合法流转路径（锁定、释放给对方、退回给自己），不存在转账给任意第三方地址的接口。
4. **人工干预终局**：CAW Pact 支持预设人类 Revoke 权限。若发生极端异常，协议拥有者可通过 CAW 控制台手动终止契约，资金原路返回，具备最终熔断能力。
5. **LLM 权限隔离**：LLM 在 AEP 中被剥夺资金裁决权。LLM 的任何输出都必须经过 Go 确定性规则的二进制校验（是/否），LLM 无法触发 CAW Recipe。
